CSV avanzado
El formato más común casi nunca viene limpio

Un CSV es texto plano: filas separadas por saltos de línea, columnas separadas por comas. La mayoría aprende read_csv() "pelado", pero en la industria un CSV real trae acentos, a veces usa ; en vez de ,, y conviene leer solo las columnas que se necesitan.

Los parámetros que más se usan

En el código del curso — 06_leer_formatos.py · leer_csv_avanzado()
df = pd.read_csv(
    ruta,
    sep=",",            # separador de columnas (a veces ';' en Chile/Excel)
    encoding="utf-8",   # cómo están codificados los acentos y la ñ
    usecols=None,       # ["col1","col2"] para leer SOLO esas columnas
    nrows=None,         # 1000 para "espiar" un archivo enorme sin abrirlo entero
)
df.head(3)            # el primer gesto al abrir cualquier dato

Encoding y separador: los dos que rompen todo

ParámetroPara qué sirveValor típico en Chile
encodingCómo se leen acentos y ñ"utf-8" o "latin-1"
sepCarácter que separa columnas"," o ";"
usecolsLeer menos columnas = menos memoria["fecha","monto"]
nrowsLeer solo las primeras N filas1000
⚠️
Error típico: UnicodeDecodeError al leer un CSV con acentos. Es el encoding equivocado. Prueba encoding="latin-1" o encoding="utf-8-sig". No es que el archivo esté "roto".

Excel con varias hojas
Los reportes corporativos casi nunca tienen una sola pestaña

Los reportes de gerencia llegan en Excel con muchas hojas (una por mes, una por sucursal). La regla de oro: no asumas qué hojas tiene — primero pregunta, después lee la que necesitas. Por detrás, pandas usa la librería openpyxl.

En el código del curso — 06_leer_formatos.py · leer_excel_multihoja()

Si sheet_name=None, pandas te devuelve un diccionario con todas las hojas a la vez: {"Ventas": df1, "Clientes": df2}. Útil cuando quieres procesarlas todas en un bucle.

JSON anidado
El idioma de las APIs: cajas dentro de cajas

¿Qué es un JSON?

Un JSON es la forma en que los sistemas web se pasan datos. En Python es exactamente un diccionario (lo que viste en la Clase 3): claves y valores, donde un valor puede ser otra caja, y dentro otra lista. Las APIs no devuelven tablas: devuelven JSON anidado. El trabajo es aplanarlo a una tabla.

json_normalize: de anidado a tabla

En el código del curso — 06_leer_formatos.py · leer_json_anidado()
# La respuesta de una API: un dict con una lista anidada adentro
respuesta_api = {
    "empresa": "Amisoft",
    "proyectos": [
        {"cliente": "Minera Norte", "sector": "minería", "monto": 50000},
        {"cliente": "Retail Sur", "sector": "retail", "monto": 32000},
    ],
}

# json_normalize "aplana" la lista anidada en una tabla ordenada
df = pd.json_normalize(respuesta_api["proyectos"])
Resultado
     cliente  sector  monto
Minera Norte minería  50000
  Retail Sur  retail  32000

Por qué importa hoy: esta sección es el puente al bloque de APIs. Cuando en la sección 7 traigamos datos de internet, llegarán como JSON exactamente así, y los aplanaremos con la misma herramienta.

Parquet
El formato del cloud: pesa menos y se lee más rápido

Un CSV guarda los datos por filas y como texto. Parquet los guarda por columnas y comprimido. ¿El resultado? El mismo dato pesa mucho menos y se lee mucho más rápido. Por eso AWS, Azure y GCP lo prefieren para datos a escala.

En el código del curso — 06_leer_formatos.py · demostrar_parquet()
# Escribir y leer Parquet es tan fácil como CSV
df.to_parquet("tabla.parquet", index=False)   # requiere pyarrow
df_leido = pd.read_parquet("tabla.parquet")

Cómo guardaPor filas, textoPor columnas, binario
TamañoMás pesadoComprimido (menos)
Velocidad de lecturaMás lentaMás rápida
¿Se abre en Excel?SíNo (es para máquinas)

Error típico en clase: Unable to find a usable engine... pyarrow. Parquet necesita el motor pyarrow. El script no se cae: avisa con un mensaje claro y te dice pip install pyarrow. Esa es la única librería nueva de la Clase 4.
💡
La ventaja de tamaño se nota con muchas filas y texto repetido. Con un dataset chico la diferencia es mínima; por eso el script genera 100.000 filas a propósito para que el ahorro sea visible.

APIs REST
Pedir datos a otro sistema por internet

¿Qué es una API? — la analogía del mozo

Una API es como el mozo de un restaurante. Tú no entras a la cocina a cocinar: le pides al mozo lo que quieres, él va a la cocina y te trae el plato. Con una API es igual: tú no entras al servidor de otra empresa; le haces un pedido ("dame el dólar de hoy") y te trae el plato (los datos, casi siempre en JSON). REST es simplemente el estilo de conversación más común para esos pedidos.

Códigos de respuesta: cómo te dice el mozo cómo le fue

CódigoSignificaEn la analogía
200OK, aquí están tus datosTe trajo el plato
404No existe lo que pedisteEse plato no está en la carta
401 / 403No autorizado / prohibidoNo tienes reserva
500Error del servidorLa cocina se incendió
💡
Cuidado: un código 200 significa que la comunicación fue bien, no que los datos sean correctos. Igual debes validarlos, como aprendiste en la Clase 3. La red puede estar OK y los datos venir vacíos.

requests en Python
La librería para hablar con APIs

El script 07_consumir_api.py consume una API real, pública y chilena —mindicador.cl— que devuelve el dólar, euro, UF y UTM del día. No pide clave. Es el caso perfecto para ver, de punta a punta, cómo internet se convierte en una tabla de pandas.

Las cuatro líneas que importan

En el código del curso — 07_consumir_api.py · consultar_indicadores()
respuesta = requests.get(URL_API, timeout=10)  # tocar la puerta
respuesta.raise_for_status()                  # si fue 404/500, error aquí
datos = respuesta.json()                    # texto JSON -> dict de Python
df = a_tabla(datos)                         # dict -> DataFrame
PiezaQué hacePor qué importa
requests.get()Pide datos (GET)Es como escribir la URL en el navegador
timeout=10Corta si no responde en 10 sNunca pidas sin timeout, o el programa cuelga para siempre
.json()Convierte la respuesta a dictDe ahí ya sabes llegar a un DataFrame

raise_for_status: convertir un error HTTP en un error de Python

Sin esta línea, una respuesta 404 o 500 pasaría como si nada y seguirías con datos vacíos. raise_for_status() lanza una excepción que puedes atrapar y manejar.

try/except: la red SIEMPRE puede fallar

En el código del curso — 07_consumir_api.py
try:
    respuesta = requests.get(URL_API, timeout=10)
    respuesta.raise_for_status()
    return respuesta.json()
except requests.exceptions.Timeout:
    logging.error("La API tardó demasiado. ¿Internet lento?")
except requests.exceptions.ConnectionError:
    logging.error("No hay conexión. ¿Estás conectado a internet?")
🛡️
Si no hay internet, el programa no explota: avisa con un mensaje claro y termina ordenado. Eso es código de producción — la misma idea de manejo de errores de la Clase 3, aplicada a la red.

Secretos y archivo .env
La clave NUNCA va en el código

La mayoría de las APIs de la industria —el SII, la CMF, SAP— piden una clave secreta. Y aquí viene el error #1 del principiante: escribir la clave directo en el código y subirla a GitHub.

El error que cuesta caro

❌ JAMÁS hagas esto
# Hardcodear la clave = dejar la llave pegada en la puerta
API_KEY = "sk-prod-9f8a7b6c5d4e3f2a1b0c"   # ¡y subirlo a GitHub!
⚠️
Caso real: en un proyecto de Amisoft, credenciales filtradas en un repositorio causaron un incidente de seguridad. Una clave en GitHub es pública para siempre, aunque la borres después.

La forma correcta: python-dotenv

import os

Reintentos con backoff exponencial

Las APIs reales se caen a veces. Un cliente profesional reintenta con paciencia creciente: espera 1 segundo, luego 2, luego 4. Eso es backoff exponencial.

En el código del curso — 08_api_segura.py
espera = 1
for intento in range(1, MAX_INTENTOS + 1):
    try:
        r = requests.get(url, headers=headers, timeout=10)
        r.raise_for_status()
        return r
    except requests.exceptions.RequestException:
        time.sleep(espera)
        espera *= 2     # 1 -> 2 -> 4  (el "backoff")
💡
¿Por qué esperar el doble? Si la API está saturada, bombardearla con pedidos idénticos empeora todo. Darle tiempo creciente le permite recuperarse, y a ti te evita que te bloqueen por exceso de pedidos (rate limiting).

Proteger datos personales (PII)
A las personas detrás de los datos se las protege siempre

¿Qué es PII?

PII (Personally Identifiable Information) es todo dato que identifica a una persona: RUT, nombre, email, teléfono, dirección. Un ingeniero de datos responsable nunca manda RUTs "en crudo" a la nube ni a una IA externa si lo puede evitar.

Hashing: seudonimizar el RUT

En el código del curso — 09_proteger_pii.py · hashear_rut()
import hashlib

def hashear_rut(rut):
    texto = (SALT + str(rut)).encode("utf-8")
    return hashlib.sha256(texto).hexdigest()[:16]

El mismo RUT siempre da el mismo código, así que todavía puedes cruzar tablas. Pero del código no se puede volver al RUT. Proteges a la persona sin perder la capacidad de analizar.

💡
Analogía: el hashing es como ponerle a cada persona una pulsera con un número de festival. Sigues sabiendo que dos compras son del mismo cliente (misma pulsera), pero no sabes su nombre real. El SALT es un texto secreto que se mezcla antes de hashear, para que nadie adivine el RUT probando combinaciones — y en un proyecto real ese salt viviría en el .env.

Enmascarar el email

En el código del curso — 09_proteger_pii.py · enmascarar_email()
def enmascarar_email(email):
    usuario, dominio = email.split("@", 1)
    return f"{usuario[0]}***@{dominio}"     # j***@gmail.com
TécnicaCuándo usarlaResultado
HashingNecesitas cruzar el dato entre tablasRUT → código irreversible y estable
EnmascararEl dato debe seguir siendo legible en un reporteemail → j***@gmail.com
⚠️
Error típico: olvidar borrar las columnas crudas después de anonimizar. El script lo hace explícito con .drop(columns=["rut", "email"]). No basta con crear la columna hasheada; hay que eliminar la original, o no protegiste nada.
10
Ley 19.628 (Chile)
El marco legal de los datos personales

En Chile, la Ley 19.628 sobre protección de la vida privada regula cómo se pueden tratar los datos personales. No necesitas ser abogado, pero sí saber que cuando trabajas con RUTs, nombres o correos, hay obligaciones legales detrás. Las técnicas de la sección 9 —hashing y enmascarado— son la forma práctica de cumplir con ese deber de cuidado.

💡
Seguridad de secretos (sección 8) y protección de datos personales (sección 9) son la misma mentalidad: tratar la información sensible con cuidado por defecto, no como un agregado al final. Esa actitud es lo que separa a un ingeniero de datos profesional de un aficionado.
