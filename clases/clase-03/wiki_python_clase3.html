Estructuras de datos
Las cuatro formas de organizar datos en Python

¿Qué es una estructura de datos?

Es una forma de organizar y guardar datos en la memoria. Python trae cuatro nativas, y cada una sirve para algo distinto. Elegir bien no es un detalle: es lo que separa un programa lento de uno rápido. Piensa en cuatro muebles de tu casa: un estante (lista), una caja fuerte (tupla), una bolsa de canicas (set) y una agenda de contactos (diccionario).

EstructuraSintaxisGarantizaCaso Olist
list[ ]Orden · mutable · duplicadosFlujo de estados de una orden
tuple( )Inmutable · hasheableHuella (order_id, estado)
set{ }Sin duplicados · búsqueda O(1)Órdenes únicas con reseña
dict{k: v}Clave→valor · acceso O(1)Mapeo de estados; una fila

Listas — secuencia ordenada y mutable

La estructura más común. Mantiene el orden de inserción, se puede modificar (mutable) y admite valores repetidos. Se accede por índice, empezando desde 0.

En el código del curso — 03_estructuras_olist.py · demo_listas()
# El flujo de estados por los que pasa una orden — el orden importa
flujo_estado = ["created", "approved", "processing", "shipped", "delivered"]

flujo_estado[0]            # "created"   → el primero (índice 0)
flujo_estado[-1]           # "delivered" → el último (índice negativo)
flujo_estado.append("invoiced")   # agrega al final (mutable)
⚠️
Error típico: el primer elemento es el índice 0, no 1. lista[1] te da el segundo elemento. Le pasa a todo el mundo al empezar.
💡
Cuándo NO usar lista: cuando solo quieres saber si un elemento existe. Buscar en una lista grande es lento (recorre uno por uno). Para eso está el set — ver sección 4.

Tuplas — inmutables y hasheables

Como una lista, pero no se puede modificar una vez creada. Eso, lejos de ser una limitación, es una protección: representa "un hecho que ya ocurrió" y no debe cambiar por accidente. Además, al ser inmutable, puede usarse como clave de un diccionario o como elemento de un set.

En el código del curso — 03_estructuras_olist.py · demo_tuplas()
# La "huella" de una orden: un par fijo que no debe cambiar
huella = ("abc123", "delivered")

oid, estado = huella       # desempaquetado: oid="abc123", estado="delivered"

conteo = {huella: 1}        # válido: una tupla SÍ puede ser clave de dict

huella[0] = "otro"          # ❌ TypeError: la tupla es inmutable
✅
Regla práctica: si el dato debería poder cambiar, usa lista [ ]. Si es un hecho fijo que no se toca, usa tupla ( ).

Sets — sin duplicados y búsqueda instantánea

Un conjunto de valores únicos. Tiene dos superpoderes: elimina duplicados automáticamente, y la búsqueda x in conjunto es instantánea (O(1)) sin importar el tamaño. Además permite operaciones de conjuntos: diferencia -, intersección &, unión |.

En el código del curso — 03_estructuras_olist.py · demo_sets()
# Convertir a set elimina duplicados al instante
ordenes_con_review = set(reviews["order_id"])
ordenes_totales    = set(orders["order_id"])

# Diferencia de conjuntos: órdenes que NUNCA tuvieron reseña — ¡una línea!
sin_review = ordenes_totales - ordenes_con_review

"abc123" in ordenes_con_review    # True/False instantáneo (O(1))
🚀
El dato de la clase: cambiar un x in lista por x in set es, literalmente, lo que en un proyecto real bajó un proceso de 8 horas a 12 minutos. La estructura correcta importa.

Diccionarios — clave → valor

La estructura más usada en datos. Asocia una clave con un valor, con acceso instantáneo (O(1)). Cada fila de una tabla, en el fondo, es un diccionario columna → dato. Usa siempre .get(clave, defecto) para no romper si la clave no existe.

En el código del curso — 03_estructuras_olist.py · demo_diccionarios()
# Mapeo de traducción de estados (clave en inglés → valor en español)
traduccion = {
    "delivered": "Entregado",
    "shipped":   "Enviado",
    "canceled":  "Cancelado",
}

traduccion.get("delivered", "desconocido")   # "Entregado"
traduccion.get("invoiced", "desconocido")    # "desconocido" → NO rompe
⚠️
Error típico: acceder con corchetes dic["clave"] asumiendo que la clave siempre existe. El día que llega un dato inesperado, lanza KeyError y el programa muere. Usa .get() con valor por defecto.

Cómo elegir la estructura
El árbol de decisión del ingeniero de datos

Ante un problema, hazte estas preguntas en orden. La primera que responda "sí" te indica la estructura.

PreguntaSi la respuesta es SÍ
¿Asocio una clave con un valor (código→nombre, id→registro)?dict
¿Solo necesito saber si algo existe, o eliminar duplicados, o comparar grupos?set
¿Es un registro fijo de pocos valores que no debe cambiar?tuple
¿Importa el orden y voy a recorrer / modificar los elementos?list
💡
El error de principiante es usar list para todo. Funciona, pero a veces es lentísimo. Cada vez que escribas "recorré la lista y fijate si ya está", detente: eso es un set pidiendo ser usado.

Comprehensions
Construir estructuras en una línea legible

¿Qué es una comprehension?

No es una estructura: es una sintaxis abreviada para construir listas, sets o diccionarios en una sola línea, reemplazando un bucle for con append. Más corto, más legible y más rápido.

List comprehension

Compara las dos formas de hacer lo mismo: filtrar pagos con tarjeta sobre R$ 500.

PYTHONforma larga vs. comprehension
# Forma larga: bucle con append (4 líneas)
altos = []
for p in pagos:
    if p["payment_type"] == "credit_card" and p["payment_value"] > 500:
        altos.append(p["payment_value"])

# Forma corta: list comprehension (1 línea, equivalente)
altos = [p["payment_value"] for p in pagos
         if p["payment_type"] == "credit_card" and p["payment_value"] > 500]

Se lee de corrido: "dame el payment_value, por cada pago, si el tipo es tarjeta y supera 500".

Set y dict comprehension

Cambiando corchetes por llaves, construyes un set (valores únicos) o un diccionario (pares clave:valor) de la misma forma.

En el código del curso — 03_estructuras_olist.py · demo_comprehensions()
# Set comprehension — tipos de pago únicos
tipos = {p["payment_type"] for p in pagos}

# Dict comprehension — conteo de reseñas por puntaje (1 a 5)
conteo = {estrella: scores.count(estrella) for estrella in range(1, 6)}
⚠️
No abuses: si la lógica tiene varios pasos o condiciones anidadas, la comprehension se vuelve ilegible. Ahí vuelve al for explícito. La legibilidad le gana a la elegancia.

Complejidad: O(n) vs O(1)
Por qué el set es tan rápido para buscar

La idea en dos analogías

Lista = pila de libros en el piso. Para encontrar uno, revisas de arriba abajo, uno por uno. Si está al fondo, los miraste todos. Cuanto más grande la pila, más tardas: eso es O(n), el tiempo crece con la cantidad.

Set / dict = biblioteca con casilleros numerados. Una fórmula (la "función hash") calcula en qué casillero va cada cosa. Para buscar, vas directo al casillero. Da igual si hay 100 o 100 millones: tarda lo mismo. Eso es O(1), tiempo constante.

OperaciónListaSet / Dict
Buscar si un elemento existe (in)O(n) — lentoO(1) — instantáneo
Eliminar duplicadosManual, lentoAutomático
Mantener ordenSíSet no; dict sí (inserción)
Acceder por posición [i]Sí, O(1)No aplica
🚀
En 100.000 elementos, buscar en lista vs. en set es la diferencia entre milisegundos y microsegundos. En millones de búsquedas, entre horas y minutos.

Control de flujo
Decidir y repetir: if, for, while

if / elif / else

Ejecuta código según una condición. elif ("else if") encadena condiciones; else es el caso restante.

En el código del curso — 04_validar_olist.py · clasificar_monto()
def clasificar_monto(valor: float) -> str:
    if valor <= 0:
        return "invalido"
    elif valor < 50:
        return "bajo"
    elif valor < 500:
        return "medio"
    else:
        return "alto"

for y while

for recorre los elementos de una colección. while repite mientras una condición sea verdadera. En datos, casi siempre usarás for.

PYTHONrecorrer filas como diccionarios
for fila in payments.to_dict("records"):
    # cada 'fila' es un diccionario {columna: valor}
    print(fila["payment_type"])

continue y break

Dos instrucciones que cambian el curso de un bucle. Son distintas y confundirlas es un error grave en validación de datos.

InstrucciónQué haceAnalogía
continueSalta al siguiente elemento, sin terminar el bucleApartar una fruta podrida y seguir la cinta
breakTermina el bucle por completoDetener la cinta entera
En el código del curso — 04_validar_olist.py · loop principal
for fila in payments.to_dict("records"):
    es_valido, motivo = validar_pago(fila)
    if not es_valido:
        rechazados.append({**fila, "motivo_rechazo": motivo})
        continue   # aparta la fila mala y sigue — NO detiene el pipeline
    validos.append(fila)
⚠️
Usar break en un validador sería un desastre: detendría todo ante la primera fila mala, dejando sin procesar miles de filas buenas. En validación casi siempre va continue.

El patrón ETL
Leer → validar → separar

ETL = Extract, Transform, Load

El esqueleto de casi todo pipeline de datos. En nuestra versión: leer los datos, validar cada fila, y separar las buenas de las malas. Como una cinta de fábrica: la fruta buena sigue a la olla, la podrida se aparta a un cajón, y la cinta nunca se detiene.

En el código del curso — 04_validar_olist.py · validar_pago()
def validar_pago(fila: dict) -> tuple[bool, str]:
    if pd.isna(fila["payment_type"]):
        return False, "tipo_nulo"
    if fila["payment_type"] not in TIPOS_VALIDOS:
        return False, f"tipo_desconocido:{fila['payment_type']}"
    if pd.isna(fila["payment_value"]) or fila["payment_value"] <= 0:
        return False, "monto_no_positivo"
    return True, "ok"
💡
El patrón "return temprano": apenas una condición falla, hace return y la función termina. Por eso usa if separados, no elif: cada return ya corta el flujo.
✅
Las filas rechazadas no se borran: se guardan con su motivo en un archivo aparte. En producción se reportan al equipo de datos. El silencio es el enemigo.

Manejo de errores
try / except / finally

¿Qué es una excepción?

Un evento que interrumpe el programa cuando algo del entorno falla: un archivo que no existe, un disco lleno, una conexión caída. El bloque try/except es el "plan B": en try va el código que podría fallar; en except, qué hacer si falla.

try / except

En el código del curso — 04_validar_olist.py · lectura robusta
try:
    payments = pd.read_csv(ruta)
    logging.info(f"{len(payments):,} pagos cargados")
except FileNotFoundError:
    logging.error(f"No se encontró {ruta}")
    raise SystemExit(1)
except pd.errors.ParserError as e:
    logging.error(f"CSV corrupto: {e}", exc_info=True)
    raise SystemExit(1)

Captura excepciones específicas

Cada except nombra el error que espera (FileNotFoundError, ParserError). Así puedes dar el plan B correcto para cada problema.

⚠️
El error más peligroso del principiante: poner except: a secas para "que no se caiga". Eso esconde los errores: el programa sigue haciendo cualquier cosa y nadie se entera. Captura siempre el error específico.

Recuperable vs fatal

SituaciónTipoQué hacer
Una fila con un dato mal escritoRecuperableApartarla con continue y seguir
El archivo completo no existeFatalraise SystemExit(1) y detener

La pregunta clave: "¿puedo seguir trabajando sin esto?" Si sí, recuperable; si no, fatal.

💡
El bloque opcional finally se ejecuta siempre, haya error o no. Sirve para cerrar recursos (archivos, conexiones). Es como apagar la luz al salir de la habitación, pase lo que pase.

Logging
La memoria del pipeline

logging vs print

La pregunta ancla

Si tu pipeline corre solo a las 3 AM y falla, ¿qué información tiene el equipo a las 8 AM? Con print: nada, la consola se cerró. Con logging: un archivo con cada evento, su hora exacta y su nivel. print es hablar en una sala vacía; logging es llevar una bitácora escrita.

En el código del curso — 04_validar_olist.py · configuración de logging
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s · %(levelname)s · %(message)s",
    handlers=[
        logging.StreamHandler(),                          # a la consola
        logging.FileHandler("data/clase3_pipeline.log"),  # y al archivo
    ],
)
logging.info(f"{len(validos):,} pagos válidos")

Niveles de severidad

No todos los mensajes son igual de graves. Como las luces del tablero del auto: verde, amarillo, rojo.

NivelCuándoEjemplo
DEBUGDetalle fino, solo al investigar"valor intermedio = 42"
INFOEvento normal, todo bien"100.000 filas cargadas"
WARNINGAlgo raro, recuperable"aparté 200 filas con nulos"
ERRORAlgo falló"no pude leer el archivo"
CRITICALCatástrofe, no se puede seguir"la base de datos no responde"
✅
Los niveles permiten alertas automáticas: "mándame un email solo si aparece un ERROR". Sin niveles, habría que leer miles de líneas a mano.

Refactorización
Mismo resultado, mejor código (con IA supervisada)

¿Qué es refactorizar?

Mejorar cómo está escrito el código sin cambiar qué hace. La IA puede generar el borrador en segundos, pero tú entiendes, revisas y firmas cada línea. Es el "vibe coding supervisado": la IA es el copiloto, tú el piloto.

Los 8 problemas del código "espagueti"

En el código del curso — 05_refactor_olist.py · versión refactorizada
UMBRAL_RESEÑA_BAJA = 3   # 1-3 estrellas = reseña negativa

def contar_resenas_negativas_con_texto(
    reviews: pd.DataFrame,
    umbral: int = UMBRAL_RESEÑA_BAJA,
) -> int:
    """Cuenta reseñas negativas que además traen comentario."""
    mask = (
        reviews["review_score"].notna()
        & (reviews["review_score"] <= umbral)
        & reviews["review_comment_message"].notna()
    )
    return int(mask.sum())
⚠️
Regla innegociable: nunca entregues código que no entiendas, lo haya escrito la IA o no. Si no lo entiendes, no puedes mantenerlo ni defenderlo. La IA acelera; entender es tu responsabilidad.

Tests con pytest
La red de seguridad del código

¿Qué es un test?

Un pequeño programa que verifica que otro funciona como debe. Como los controles de calidad en una fábrica de autos: ¿frenan los frenos? ¿encienden las luces? Si un control falla, el auto no sale. Los tests te dan confianza para cambiar el código sin miedo: modificas, corres los tests, y si pasan en verde, no rompiste nada.

assert

La base de todo test. assert afirma que algo es verdadero; si no lo es, falla y avisa.

PYTHONassert
assert clasificar_resena(1) == "negativa"   # pasa si es verdadero
assert res_a == res_b, "¡Los resultados difieren!"  # con mensaje de error

Anatomía de un test

En el código del curso — test_clase3.py
def test_una_estrella_es_negativa():
    assert clasificar_resena(1) == "negativa"

def test_tres_estrellas_es_neutra():
    assert clasificar_resena(3) == "neutra"

def test_cinco_estrellas_es_positiva():
    assert clasificar_resena(5) == "positiva"

Un buen test no solo detecta el error: te dice qué esperaba y qué obtuvo, señalando dónde está el problema. Es un diagnóstico, no solo una alarma.
